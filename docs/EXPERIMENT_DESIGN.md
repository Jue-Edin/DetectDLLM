# Experiment design

## Goal

Detect whether a text is human-written or generated by the local DUO DLLM.

## Data construction

1. Start from a human corpus in `data/human/`.
2. Split the source passages into train/val/test by hashing `source_id` with a fixed seed.
3. For every selected human passage, generate one DUO passage with matched target length.
4. Evaluate all detectors on the resulting balanced binary dataset.

## Prompt meaning in this repository

In this repository, `prompt` means a **human text prefix** used to condition DUO.
It does **not** mean a chat-style instruction such as
`Please rewrite the following passage...`.

For the prompt-paired pilot, the intended construction is:

- tokenize a human passage with the DUO tokenizer;
- keep the first `prompt_tokens` tokens as the prefix;
- let DUO generate the remaining tokens to a fixed total length.

This keeps the data construction simple and controls nuisance variables such as
prompt style, topic drift, and length mismatch.

## Three experiments reported in this repo

### Experiment A: proposed method

- detector: `our_duo_analytic`
- scoring model: DUO
- reference model: DUO
- score: analytic normalized masked-token score

This is the main DLLM-specific detector.

### Experiment B: Fast-DetectGPT surrogate baseline

- detector: `fastdetectgpt_surrogate`
- scoring/reference model: a local AR model such as GPT-2
- evaluated on: the **same DUO-generated texts** used in A and C

Important: GPT-2 is only the surrogate model used by the baseline detector.
The machine texts themselves are still generated by DUO.

### Experiment C: DUO plain baseline

- detector: `duo_plain_meanlogprob`
- scoring model: DUO
- reference model: DUO
- score: plain masked-token mean log-probability

This is the most direct same-source ablation for measuring how much the analytic
normalization helps.

## Validation protocol

- Choose score orientation on validation only.
- Choose threshold on validation only.
- Choose the best mask ratio for the DUO detectors using validation ROC-AUC only.
- Report final metrics on test.

## Why there is a train/val/test split

For the current zero-shot detectors:

- **train** is not used to fit model weights;
- **val** is used for model selection and calibration;
- **test** is used for the final held-out report.

Concretely, the current code uses **validation** to choose:

- score sign/orientation;
- threshold;
- best mask ratio.

So the crucial split is really **val vs test**.
The train split is kept mainly for consistency, future extensions, and possible
later additions such as supervised calibrators or learned ensembles.

## Metrics

- ROC-AUC
- PR-AUC
- Accuracy
- Balanced accuracy
- TPR@1%FPR
- TPR@5%FPR
- TPR@10%FPR
